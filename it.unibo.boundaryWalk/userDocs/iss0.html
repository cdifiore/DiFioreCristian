Introduction

Our motto:
there is no code without a project, no project without problem analysis and no problem without requirements.

Requirements

Design and build a software system that leads a robot to walk along the boundary of a empty, rectangular room.

Requirement analysis

The interaction with the customer clarified the meaning of the names and of the verbs in the requirement text.
About the names:
  • room: conventional room
  • boundary: the perimeter of the room which is surrounded by walls
  • robot: a device capable of moving by receiving commands using the network VirtualRobot2021.html.
Robot's action:
  • walk: the robot has to move forward along the room walls

The first user story

After put the robot in the HOME cell (south direction) I turn on the system that makes move robot using wifi commands.

The system cannot terminate without the robot's task is completed. As a user I can't stop the execution and the system will stop itself.

At the end of execution I expect that the robot has completed the path.

Verification of expected results (Test plans)

It is necessary to verify that the path is followed correctly
It must be possible to verify the path through software without human intervention.

Problem analysis

Relevant Aspect

  1. the distributed system to be implemented consists of two macro-components:
    • the robot given by the customer
    • our application ( boundaryWalk ) that sends commands to the robot in order to meet the requirements
  2. The robot can be controlled in two different ways(using network), as described in VirtualRobot2021.html: commands:
    • sending messages to the port 8090 with HTTP POST protocol
    • sending messages to the port 8091 using websocket
  3. There isn't an abstraction-gap on the operation level thanks to the libraries in programming languages that allow to send commands.
    However, there is a abstraction gap on a conceptual level : there are two possible technologies for the communication with the robot and each requires appropriate supports. On a conceptual level is always used a request-response interaction scheme.
  4. We should be able to be built the prototype in three working days.

Problems identified

  1. Interaction abstraction

    The software system should be made as independent as possible from the communication protocol used for the interaction with WEnv.
    The designer could use a design pattern to make the application layer as invariant as possible for the communication protocol.

  2. Testing

    The main goal is to define an automated TestPlan that requires you to infer knowledge from WENV's previous commands and responses.
    It is necessary to create within the application a sort of 'mind map' capable of providing a model of the robot's actions, on which a testing program can try some verifications.

    The following algorithm can be used to obtain the verification activity: 
    the robot starts from the HOME position, facing south (DOWN) for 4 times:
	At first-> send  moveForward commands until the robot hits the opposite wall
	Then-> send a turnLeft rotation command

TestPlan strategies: 'infer knowledge' from commands

We introduce the following abbreviations related to commands cril: 
w :  {"robotmove":"moveForward", "time": 600}
s :  {"robotmove":"moveForward", "time ":600}
h :  {"robotmove":"moveForward", "time ":100}
l :  {"robotmove":"turnLeft ",   "time ":300}
r :  {"robotmove":"turnRight",   "time ":300}
Chosen TestPlan:
  • The time for the moves "w" and "s" is set in such a way as to obtain movements of robot-units with which we incrementally build a map of the territory after each move. In this way you can know where the robot is after each command and therefore you can check the path taken at the end of the application. This approach provides a high level of knowledge that makes the application scalable.
